# fos-gateway Documentation

## Purpose of this folder

`fos-gateway` is the single entry point for backend HTTP traffic in this repository.

It is responsible for:

- validating incoming JWTs (Keycloak JWKS)
- enforcing gateway-level security
- adding cross-cutting headers (`X-FOS-Request-Id`, `X-FOS-Actor-Id`)
- rate limiting requests (Redis-backed)
- routing requests to downstream services (`fos-governance-service` and `fos-workspace-service`)

In short: this module protects and standardizes requests before they reach domain services.

## Folder contents

### Root files

- `fos-gateway/pom.xml`
  - Maven module definition.
  - Declares Spring Cloud Gateway, OAuth2 resource server, Redis reactive, actuator, and test dependencies.

- `fos-gateway/README.md`
  - This documentation.

- `fos-gateway/.project`, `fos-gateway/.classpath`, `fos-gateway/.settings/*`
  - IDE metadata (Eclipse). Not part of runtime behavior.

- `fos-gateway/target/*`
  - Build output (compiled classes, surefire reports, jar). Auto-generated by Maven.

### Runtime source files (`src/main`)

- `fos-gateway/src/main/java/com/fos/gateway/GatewayApp.java`
  - Spring Boot entry point (`main`).
  - Starts the reactive gateway application.

- `fos-gateway/src/main/java/com/fos/gateway/config/GatewayRoutesConfig.java`
  - Central route map.
  - Sends governance endpoints (`/api/v1/actors`, `/players`, `/teams`, `/policy`, `/signals`, `/identity`) to `fos.governance.url`.
  - Sends workspace endpoints (`/api/v1/documents`, `/events`, `/profiles`, `/notifications`, `/search`, `/onlyoffice`) to `fos.workspace.url`.

- `fos-gateway/src/main/java/com/fos/gateway/config/SecurityConfig.java`
  - Configures WebFlux security filter chain.
  - When `fos.security.enabled=true`: all routes require JWT except `/actuator/health`.
  - When disabled: allows all requests (useful for local tests/dev).

- `fos-gateway/src/main/java/com/fos/gateway/filter/CorrelationIdFilter.java`
  - Global filter that ensures `X-FOS-Request-Id` exists on every request.
  - If missing, generates a UUID and forwards it.

- `fos-gateway/src/main/java/com/fos/gateway/filter/ActorIdEnrichmentFilter.java`
  - Global filter that extracts JWT subject and sets `X-FOS-Actor-Id`.
  - Gives downstream services a normalized actor header without re-parsing JWT at the edge.

- `fos-gateway/src/main/java/com/fos/gateway/config/RateLimiterConfig.java`
  - Defines `actorKeyResolver` bean for rate limiting keys.
  - Uses `X-FOS-Actor-Id` first, then fallback to remote IP if unavailable.

- `fos-gateway/src/main/resources/application.yml`
  - Main runtime configuration:
    - server port (`8080`)
    - JWT JWKS URI
    - Redis host/port
    - default `RequestRateLimiter` settings
    - service URLs (`fos.governance.url`, `fos.workspace.url`)

### Test files (`src/test`)

- `fos-gateway/src/test/java/com/fos/gateway/GatewayJwtTest.java`
  - Verifies JWT behavior:
    - valid JWT request succeeds
    - missing JWT returns `401`
    - health endpoint is public
    - forwarded request includes `X-FOS-Actor-Id` and `X-FOS-Request-Id`

- `fos-gateway/src/test/java/com/fos/gateway/CorrelationIdFilterTest.java`
  - Verifies correlation header behavior:
    - generated if absent
    - preserved if already provided by caller

- `fos-gateway/src/test/java/com/fos/gateway/GatewayRateLimitTest.java`
  - Verifies Redis-backed rate limiting with Testcontainers Redis:
    - first request allowed
    - immediate second request throttled (`429`)

## How the files work together

Request path through this module:

1. `GatewayApp` boots Spring Gateway.
2. `SecurityConfig` authenticates JWT (unless security is disabled).
3. `CorrelationIdFilter` ensures trace ID header.
4. `ActorIdEnrichmentFilter` injects actor header from JWT.
5. Rate limiter (configured in `application.yml`, keyed by `RateLimiterConfig.actorKeyResolver`) enforces per-actor quotas.
6. `GatewayRoutesConfig` forwards request to governance or workspace service.

So each component has a narrow role: security, observability, identity propagation, traffic protection, then routing.

## How this folder interacts with other backend modules

- Interacts with `fos-governance-service`
  - Receives routed requests for identity/canonical/policy/signal APIs.
  - Governance then handles domain logic.

- Interacts with `fos-workspace-service`
  - Receives routed requests for documents, events, profiles, notifications, search, and onlyoffice APIs.
  - Workspace then handles document/event/profile workflows.

- Interacts with infrastructure services
  - Keycloak JWKS endpoint: JWT signature verification.
  - Redis: distributed rate limiting counters.

The gateway is intentionally thin: it does not own domain data; it enforces edge concerns and forwards traffic.

## Operational notes

- Main local endpoint: `http://localhost:8080`
- Security toggle: `FOS_SECURITY_ENABLED`
- Downstream URLs: `GOVERNANCE_URL`, `WORKSPACE_URL`
- JWKS source: `KEYCLOAK_JWKS_URI`
- Redis host/port: `REDIS_HOST`, `REDIS_PORT`
