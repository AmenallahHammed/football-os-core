\chapter{Design}

\section*{Introduction}
\addcontentsline{toc}{section}{Introduction}

This chapter presents the design of Football OS and answers how the platform is
organized and how the main processes are implemented. It focuses on the global
architecture, backend module design, data design, processing design, and
security and authorization design.

\section{Global Architecture Design}

\subsection{Architectural Style}

Football OS follows a Service-Oriented Architecture (SOA). The platform is
organized into services with clear responsibilities. The API Gateway is the
entry point for routing and secure access. The Governance Service handles
users, roles, permissions, canonical players and teams, and audit. The
Workspace Service handles documents, calendar/events, notifications, search,
and player profiles. The Shared SDK provides common reusable backend
mechanisms.

Backend services also follow Onion Architecture principles. Domain and business
logic remain at the center. Application logic coordinates use cases. Technical
infrastructure stays outside the core logic. External technologies are accessed
through adapters or abstractions, including MongoDB, PostgreSQL, MinIO, Kafka,
OPA, Keycloak, and OnlyOffice.

The Shared SDK supports this Onion design by offering common abstractions such
as security context, policy client, canonical resolver, storage port, event
contracts, and audit/testing utilities.

\section{Backend Module Design}

\subsection{API Gateway}

The API Gateway is the entry point of the backend platform. It routes API
requests from the frontend to the appropriate backend service. It also supports
the authentication and security flow by forwarding authenticated user context.
This design hides internal service addresses from the frontend and simplifies
external access.

\subsection{Governance Service}

The Governance Service manages actors and roles and maintains canonical
football data such as players and teams. It also supports authorization by
providing policy decisions through the governance/policy flow. In addition, it
records audit and activity information for traceability and exposes canonical
data to other services.

\subsection{Workspace Service}

The Workspace Service implements the daily operational features of the
platform. It manages documents, events, notifications, search, and player
profiles. Workspace aggregates are stored in MongoDB. The service integrates
with Governance for policy and canonical data, with MinIO through the SDK
storage abstraction, and with OnlyOffice for online document view and edit
flows.

\subsection{Shared SDK}

The Shared SDK is not a user-facing module. It is reused by backend services to
avoid duplicated technical plumbing. It provides shared security context,
policy client, canonical resolver, storage port abstraction, event contracts,
audit hooks, and test utilities.

The flow is organized as follows:
\begin{enumerate}
    \item The user starts a document upload or opens a document.
    \item The frontend sends the request through the Gateway.
    \item The Workspace Service checks authorization through Governance/Policy.
    \item For upload, Workspace generates an upload URL through StoragePort.
    \item The file content is stored in MinIO.
    \item Metadata and version information are stored in MongoDB.
    \item For online viewing/editing, OnlyOffice opens the document.
    \item If edited, a new version is stored and metadata is updated.
\end{enumerate}

The flow is organized as follows:
\begin{enumerate}
    \item An authorized user creates, edits, or deletes an event.
    \item The request passes through the Gateway.
    \item The Workspace Service checks authorization.
    \item Event data is validated.
    \item The event is stored or archived in MongoDB.
    \item Tasks, attendees, required documents, and notes are saved when provided.
    \item Notifications are generated for concerned users.
    \item Important actions are tracked for traceability.
\end{enumerate}

In this design, delete operations are implemented as archive actions rather
than hard deletion.

This flow describes how the platform builds a secured player profile.

The flow is organized as follows:
\begin{enumerate}
    \item The user opens a player profile.
    \item The frontend sends the request through the Gateway.
    \item The Workspace Service reads the authenticated user context.
    \item Workspace loads canonical player identity through the Canonical Resolver.
    \item Workspace checks which tabs the user may access.
    \item Workspace loads active player-linked documents from MongoDB.
    \item Documents are grouped by category.
    \item Temporary download URLs are generated for visible documents.
    \item The response contains only authorized sections.
\end{enumerate}

\section{Security and Authorization Design}

Authentication is handled through the identity provider. The Gateway and
backend services rely on the authenticated user context to process requests.
Authorization is role-based and policy-driven. Workspace calls Governance/Policy
to verify whether an action is allowed, and OPA evaluates the policy rules.
Sensitive resources, especially medical documents, are strongly protected.
Important actions are tracked to ensure auditability.

\begin{table}[H]
\centering
\caption{Authorization design examples}
\begin{tabular}{|p{6.2cm}|p{7.6cm}|}
\hline
\textbf{Action} & \textbf{Authorized roles / rule} \\
\hline
Create, edit, delete event & Head Coach and Club Administrator according to backend policy \\
\hline
Read events & Coaching staff roles and Club Administrator \\
\hline
Access normal documents & Authorized club staff according to policy \\
\hline
Access medical documents & Medical Staff and explicitly authorized roles \\
\hline
Access admin/contract documents & Club Administrator \\
\hline
Manage users and roles & Club Administrator \\

\end{table}
